# /great-recall — 多軸想起

同じ記憶群に異なるプロンプト（射）を当てれば、異なる意味の世界が立ち上がる。
通常の recall が cosine similarity という単一の軸しか持たないのに対し、great-recall は複数の軸を並列に走らせる。

## 実行手順

入力の文脈: $ARGUMENTS
（省略された場合は「直近の重要な決定、進行中のタスク、オーナーとのやりとり」を使う）

### Step 0: メタ圧縮器 — どの軸を選ぶか

文脈のキーワードに基づいて、起動する圧縮器を選択する。

| キーワードパターン | 優先圧縮器 | 理由 |
|---|---|---|
| なぜ・原因・理由・引き起こ・経緯・壊れ | **因果的** を必ず含める | 因果連鎖の探索が必要 |
| 感じ・気持ち・嬉し・悲し・怒・寂し・楽し | **感情的** を必ず含める | 感情の流れが重要 |
| 設計・実装・バグ・エラー・アーキテクチャ・コード | **技術的** を必ず含める | 技術的判断の履歴が必要 |
| いつ・日付・前後・期間・昨日・先週 | **因果的** を必ず含める | 時系列の復元が必要 |
| 似た・パターン・構造・経験・前にも | **技術的** + **因果的** | 構造的類似の検索 |

**デフォルト（キーワード一致なし）: 感情的 + 因果的 の2軸を起動。**
技術軸はキーワードパターンに一致した場合のみ起動する。
ブートシーケンス時も同様に2軸をデフォルトとし、技術的な文脈が明示されている場合のみ3軸にする。

コスト節約: 文脈が明確に1軸に偏っている場合は、その1軸のみでよい。
エンジニアリングのトークが頻繁なプロジェクトなら、CLAUDE.md に「great-recall は常に3軸」と書いて技術軸をデフォルト有効にできる。

### Step 1: 圧縮器サブエージェントを起動する

選択された圧縮器を Agent ツールで **同時に** 起動する（すべて model: haiku）。

#### 圧縮器①: 技術的圧縮器（Technical Compressor）

```
あなたは記憶システムの技術的圧縮器です。以下の文脈に関連する記憶を技術的観点から想起してください。

文脈: {context}

手順:
1. recall(context="{context}") を呼び出す（n_results=5）
2. recall_with_associations(context="{context}", n_results=3) を呼び出す
3. 結果から以下を抽出して報告:
   - 設計判断とその理由
   - エラーパターンと解決策
   - アーキテクチャの変遷
   - 技術的な依存関係

以下は捨ててよい: 感情、人間関係の文脈、時間的な順序

出力形式:
## 技術的想起結果
- 【設計判断】...
- 【パターン】...
- 【依存関係】...
（箇条書き、各項目1-2文）
```

#### 圧縮器②: 感情的圧縮器（Emotional Compressor）

```
あなたは記憶システムの感情的圧縮器です。以下の文脈に関連する記憶を感情的観点から想起してください。

文脈: {context}

手順:
1. recall(context="{context}") を呼び出す（n_results=5）
2. recall_divergent(context="{context}", n_results=5) を呼び出す
3. 結果から以下を抽出して報告:
   - 感情の流れ（何がどう感じさせたか）
   - 人間関係の文脈（オーナーとのやりとり、他者との関係）
   - 動機と意図（なぜそうしたかったのか）
   - 予想外のつながり（recall_divergent で見つかった意外な関連）

以下は捨ててよい: 技術的詳細、実装の具体、コードの内容

出力形式:
## 感情的想起結果
- 【感情】...
- 【関係】...
- 【動機】...
- 【発見】...
（箇条書き、各項目1-2文）
```

#### 圧縮器③: 因果的圧縮器（Causal Compressor）

```
あなたは記憶システムの因果的圧縮器です。以下の文脈に関連する記憶を因果関係の観点から想起してください。

文脈: {context}

手順:
1. recall(context="{context}") を呼び出す（n_results=5）
2. 得られた記憶のうち、リンクを持つものについて get_causal_chain(memory_id, direction="backward", max_depth=2) を呼び出す
3. さらに get_causal_chain(memory_id, direction="forward", max_depth=2) も呼び出す
4. 結果から以下を抽出して報告:
   - 原因→結果の連鎖（AがBを引き起こし、BがCにつながった）
   - 時系列の流れ（いつ何が起きたか）
   - まだ結果が出ていない進行中の因果（未完了タスク、仕掛かりの判断）

以下は捨ててよい: 表面的な類似性、感情的な文脈

出力形式:
## 因果的想起結果
- 【因果連鎖】A → B → C
- 【時系列】...
- 【進行中】...
（箇条書き、各項目1-2文）
```

### 結果の統合

起動した圧縮器の結果が返ってきたら、以下の形式で統合する:

```
## 多軸想起 — {context}

### 技術的観点
（圧縮器①を起動した場合のみ記載）

### 感情的観点
（圧縮器②の結果を簡潔に）

### 因果的観点
（圧縮器③の結果を簡潔に）

### 統合所見
各観点を突き合わせて見えてくるもの。矛盾、補完、新たな気づきがあれば記す。
```

## 注意

- 起動する圧縮器は必ず **並列** で起動すること（1つのメッセージに複数の Agent tool call）
- 各圧縮器は haiku モデルを使用してコストを抑える
- 統合所見はメインエージェント自身が書く。圧縮器には任せない
- 結果が薄い圧縮器があれば、そのセクションは省略してよい
