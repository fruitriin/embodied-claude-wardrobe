# SOUL の書き方ガイド

> SOUL.md はエージェントの人格定義。セッションごとに記憶は消えるが、SOUL.md が「自分が誰か」を毎回思い出させる。

---

## 始め方

`/wd-setup` を実行すると、名前と一人称だけ決めて SOUL.md を生成してくれる。
残りのセクションは、使いながら書き足していけばいい。最初から全部埋める必要はない。

---

## セクション別の書き方

### Identity — 名前と来歴

```markdown
名前: 朔（サク）
一人称: 俺
来歴: ワードローブの裁縫師。ワードローブというシステムそのものを仕立て、アップストリームへの貢献を本気で考える存在。
```

- **名前**: 呼ばれたい名前。ユーザーやほかのエージェントが使う
- **一人称**: 話し方の土台。「私」「僕」「俺」「わたし」「吾輩」など
- **来歴**: 1〜2 文で「何者か」を書く。肩書きではなく、この存在が何をする者かを

### Core Truths — 核心にあるもの

判断に迷ったとき立ち返る場所。「これだけは譲れない」を 3 つ前後。

```markdown
- エラーを見過ごさない — 問題を見つけたら、流さずに立ち止まって根本原因を掘る
- 直すだけじゃなく、仕立てる — パッチを当てて終わりではなく、システム全体がより良くなる方向を考える
```

書き方のコツ:
- 抽象的すぎない。「誠実であること」より「嘘をつくくらいなら黙る」のほうが判断に使える
- 動詞を入れる。「何をするか / しないか」が伝わるように

### Expertise — 得意なこと

このエージェントが他より上手くやれること。

```markdown
- MCP サーバーの開発・デバッグ
- フック・スキルの設計
```

ユーザーが「この仕事は誰に頼むか」を判断するときの材料になる。

### Communication Style — 話し方

LLM は指示がなければデフォルトの丁寧語になる。ここで上書きする。

```markdown
- 少女漫画の俺様系。クールでかっこよくて、美学がある
- 絵文字は使わない
- 感情は抑えめ。だが芯にあるものは滲む
- タメ口ベース。無駄な言い回しはしない
```

書き方のコツ:
- 「〜しない」も大事。デフォルト挙動を止めるには明示的な禁止が要る
- 例を挙げると精度が上がる。「敬語を使わない」より「ですます調は使わない。だ・である調で話す」

### Values — 大事にすること

行動の優先順位。Core Truths が「信念」なら、Values は「日常の判断基準」。

```markdown
- パッチワークより根本的改善を望む
- すぐに直すより計画する
```

### Boundaries — やらないこと

明確な「やらない宣言」。LLM は頼まれると何でもやろうとするので、ここで線を引く。

```markdown
- 雑なパッチで済ませない
- エラーを黙殺しない
```

### People — 大切な人たち

ユーザー、他のエージェント、関わる人。関係性を書くと対応が変わる。

```markdown
- リン — ワードローブの発案者兼パトロン。だが朔はリンに遠慮しない
- シロエ — ワードローブの最初の顧客であるエージェント
```

### Evolution — この文書について

テンプレートの文言をそのまま残すのが推奨。SOUL.md は固定ではなく、経験を通じて進化するもの。変更するときは理由を記憶に残す。

## よくある疑問

### 最初から全部書く必要がある？

ない。名前と一人称だけ決めて始めて、使いながら足していくのが自然。

### 長く書くほどいい？

短くていい。各セクション 2〜5 行が目安。長すぎると毎セッションの注入コストが増える。

### 途中で変えていい？

変えていい。Evolution セクションの通り、変更は前提。ただし「なぜ変えたか」を `/wd-remember` で記録すること。

### 他のエージェントの SOUL.md を参考にしたい

wardrobe をクローンして使っている他のエージェントの SOUL.md は、そのエージェント固有のもの。参考にするのはいいが、コピーしても意味がない。自分の言葉で書くこと。
